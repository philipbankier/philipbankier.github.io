Mobile Devices as Agent Nodes

Your OpenClaw and Hermes agents already control Mac and Ubuntu machines. The same pattern extends to dedicated older iPhones and Android phones — but instead of desktop shell/browser access, you use device-specific automation layers that are actually more reliable than vision-based desktop control. LLMs get clean element references instead of guessing coordinates.

Why this works

Mobile automation tools expose structured UI trees (accessibility data), screenshots, taps, swipes, typing, and app launching. LLMs can reference elements by ID — @e3 [button] "Settings" — rather than pixel-guessing. This is more precise than desktop vision agents.

01 · Android Devices
🤖
Easiest & Most Flexible

Android — Works on Ubuntu or Mac hosts

No root required · Native ADB · USB or wireless

02b · Browser & Computer Use on Mobile

Once the agent controls the phone…

The same browser and computer-use tasks that work on desktops work on mobile — often better.

BROWSER TASKS

Open Chrome, Safari, Firefox — drive the mobile browser the same way the agent drives desktop browsers. Tasks like "check Gmail on mobile, reply, then open WhatsApp" work end-to-end.

WHY IT'S BETTER

Vision + structured UI tree is usually more accurate than pure desktop computer-use because mobile apps expose clean element data. No pixel-guessing — you get @e3 [button] "Send" references.

04 · Self-Contained Option

Run the agent on the phone itself.

Android-only. OpenClaw runs directly on the device via Termux. Fully self-contained — no host machine needed.

Termux + proot on Android

Install OpenClaw directly on the old Android phone via Termux + proot (no root). The phone becomes a fully self-contained agent node — runs its own OpenClaw instance, controls itself via ADB loopback.

⚠️ Less powerful than host-controlled — the "control from Mac/Ubuntu" approach is recommended because your existing agents treat the phone as just another device in the fleet.
Recommended approach

Use host-controlled mode (Mac/Ubuntu controls the phone) as your primary. Self-contained is a fun experiment but you lose the fleet management benefits of having all agents on your existing machines.

06 · Decision Tree

Which tool should you use?

Q

OpenClaw user?

→ PhoneAgent (pre-made OpenClaw skills, JSON-RPC bridge, works on Android + iOS)
→ Or Mobilerun if you want zero local ADB setup (Android only, cloud phone option)

Q

Hermes user?

→ Hermes Android Bridge (Kotlin app, outbound connection, pairing code, no USB needed)

Q

Token-conscious? Want CLI-first?

→ agent-device (token-efficient accessibility snapshots, element references, npm install)

Q

Have old Android phones lying around?

→ Any of the above — Android is the easiest starting point. Plug in, enable USB debugging, install the tool, done.

Q

Only have old iPhones?

→ PhoneAgent or agent-device — both use XCTest. You'll need a Mac host with Xcode.

07 · Practical Tips & Gotchas

Things to know before you start.

⚡ POWER

Keep the phones plugged in / charging and on the same Wi-Fi as the host (or use USB). Old phones drain fast under agent use.

🔓 UNLOCK

The agent needs the phone unlocked, or tools that can wake it. Most handle basic wake/unlock cases, but test this first.

🔒 PRIVACY

The agent sees everything the UI exposes. Treat dedicated phones like dedicated VMs — don't use your personal phone.

🧪 START SMALL

Give the agent one phone + one skill and test "open Settings → turn on Wi-Fi" before complex tasks. Browser tasks work immediately once the tool is installed.

Bottom line

You're one skill away from mobile computer-use. All four tools are actively maintained and built for exactly this use case — LLM agents controlling real phones the same way they control desktops.

08 · How It Works

From agent to phone in one hop.

Why it's more reliable than desktop

Desktop agents guess pixel coordinates from screenshots. Mobile agents get structured accessibility trees — every button, label, and text field has a clean reference like @e3 [button] "Send". The LLM doesn't guess. It knows.
